# 🖥️ realtime-push-simulator

**浏览器端高频实时数据推送渲染方案对比演示**

在线对比「直接渲染」和「四层削峰优化」两种方案在高频数据推送场景下的表现差异。

## 🎯 核心对比

| 方案 | 策略 | 帧率 | 说明 |
|------|------|------|------|
| ❌ 直接渲染 | 每次收到数据立即渲染 | 5~15 FPS | 主线程阻塞，卡顿严重 |
| ✅ 四层削峰 | WebWorker + Map缓存 + 16ms节流 + rAF | 60 FPS | 平滑流畅 |

## 🏗️ 优化方案架构

```
高频数据推送 ──→ WebWorker(Map缓存去重) ──→ 16ms节流 ──→ rAF渲染
 (300实体×100x)      (同ID自动覆盖)        (削峰2000x)    (60fps平滑)
```

## 🚀 快速开始

```bash
npm install
npm run dev
```

打开浏览器访问 `http://localhost:5173`

## 🎮 操作说明

- 拖动「实体数量」滑块：调整模拟的实体个数（50~500）
- 拖动「推送倍速」滑块：调整数据推送倍速（10x~200x）
- 点击「开始推演」：启动模拟，左右两侧同时运行
- 观察 FPS 计数器：对比优化前后的帧率差异

## 🛠️ 技术栈

- **前端框架**：React 18 + TypeScript
- **构建工具**：Vite 5
- **渲染引擎**：HTML5 Canvas 2D
- **架构模拟**：内存 WebWorker 模拟（Map 缓存 + 16ms 定时推送）

## 💡 设计思路

本项目模拟了真实项目中的高频实时数据渲染方案：

1. **模拟推送层**：内存中模拟 gRPC Streaming 高频数据推送，支持自定义实体数量和推送倍速
2. **WebWorker 模拟**：在主线程中模拟 Worker 的 Map 缓存行为，同 ID 实体自动覆盖，只保留最新状态
3. **节流层**：16ms 定时刷新（对齐 60fps），将高频推送削峰 2000 倍
4. **渲染层**：rAF (requestAnimationFrame) 驱动 Canvas 渲染，保证平滑 60fps

> ⚠️ 注意：为了在浏览器中直接运行，本项目用内存模拟替代了真实的 gRPC + WebWorker + postMessage 链路。实际项目中这四层分别运行在独立线程/进程中，削峰效果更显著。
